# Graytail 发布就绪、战斗解耦与产品化设计

## 目标

本轮改造把 Graytail 从“编辑器和命令行可验证的原型”推进到“打包后可直接游玩、异常退出不破坏规则、战斗结果不依赖 UI 和帧率、核心玩法数值可配置”的发布候选状态。

设计遵循以下原则：

1. 玩家启动程序后必须直接看到主菜单，不依赖控制台命令。
2. Shipping 构建不包含玩家可访问的作弊入口。
3. 所有影响永久进度的操作必须先可靠落盘，再向玩家宣告成功。
4. 异常退出不能成为规避装备损失或重刷地图的手段。
5. 游戏规则属于模拟内核，UMG 只负责输入转发和表现。
6. 同一 Seed、相同输入序列必须得到相同战斗结果，不受渲染帧率影响。
7. 玩法数值来自 JSON；颜色、布局和纯视觉动画参数留在表现层。
8. 自动验证必须覆盖真实打包入口，而不只验证进程能够启动。

## 范围拆分

改造分成三个堆叠分支和草稿 PR。每个 PR 必须独立完成构建、冒烟和对应打包验证。

### PR 1：发布入口与存档安全

分支：`codex/release-entry-save`

负责：

- 打包程序自动创建 HUD 和主菜单。
- Shipping 作弊隔离。
- 存档版本升级、备份恢复和错误提示。
- 未完成局押注凭证。
- Meta 状态事务化提交。
- Development 打包入口探针。

### PR 2：独立战斗模拟器

分支：`codex/combat-simulation`

基于 PR 1，负责：

- 固定时间步战斗模拟。
- 玩家和怪物战斗位置、AI、攻击阶段、弹道与伤害判定。
- 显式暂停状态。
- Seed 派生随机流。
- 轻量只读战斗快照。
- 帧率一致性和扫掠碰撞测试。

### PR 3：产品化与自动验证

分支：`codex/productization-tests`

基于 PR 2，负责：

- 剩余玩法数值 JSON 化。
- 键鼠按键重映射。
- 中文文本接入 UE 本地化系统。
- 过期文档更新。
- 一键发行验证脚本。
- 轻量 GitHub CI 和完整发布检查清单。

## PR 1 设计

### 正式启动入口

新增 `AGT_GameMode` 和 `AGT_PlayerController`。

`AGT_PlayerController::BeginPlay` 在本地玩家存在时创建唯一的 `UGT_GameHudWidget`，添加到 viewport，显示鼠标并设置 UI 输入模式。重复进入 `BeginPlay` 或 PIE 重启不得创建第二份 HUD。

`DefaultEngine.ini` 使用 Graytail GameMode，不再使用引擎 `GameModeBase`。游戏仍可使用轻量 Entry 世界，因为玩法画面由全屏 UMG 提供，不新增无必要的 3D 地图资产。

`gt.HUD` 保留为非 Shipping 调试命令，但不再是产品入口。

### Shipping 作弊隔离

作弊能力分成“产品必需查询接口”和“开发命令”两类：

- UI 读取运行状态改走正式 Query/Run 接口，不再依赖名为 Debug 的入口。
- GM 加币、传送、无敌、强制怪物、重置存档等功能使用 `#if !UE_BUILD_SHIPPING` 编译隔离。
- 设置界面的作弊开关和提示在 Shipping 中不构建。
- 调试控制台命令注册在 Shipping 中不编译。
- Shipping 自动检查二进制行为：设置页无作弊控件，暂停页无作弊入口。

不把“默认关闭”视为安全边界。

### 存档版本 2

`UGT_MetaSaveGame::SaveVersion` 从 1 升级到 2。

版本 2 新增 `FGT_ActiveRunEscrow`：

- `bActive`
- `RunId`
- `Seed`
- `Difficulty`
- `EquippedItemIds`
- `ConsumedConsumables`
- `StartedAtUtc`

版本 1 读入后迁移为版本 2，押注凭证默认为未激活。未知的更高版本拒绝载入，不用空状态覆盖。

### 未完成局状态机

开局流程：

1. 校验游戏数据和当前 Meta 状态。
2. 在候选状态中扣除带入消耗品。
3. 记录当前装备、Seed、难度和 RunId 到激活的押注凭证。
4. 一次性提交候选状态。
5. 只有提交成功才创建 `UGT_RunContext` 并发布 `RunStarted`。

正常撤离：

1. 从运行态和押注凭证计算完整候选 Meta 状态。
2. 返还未使用消耗品，加入金币与回收物，更新统计。
3. 清除押注凭证。
4. 一次性提交。
5. 提交成功后标记本局已结算。

如果第 4 步失败，运行上下文保留终局结果，押注凭证保持激活，界面进入“结算保存失败”状态并提供重试。保存成功前不得开始新局、返回标题或把奖励显示为已经永久到账。若存储始终不可写，玩家必须收到明确提示；系统不能伪造一次成功结算。

正常失败或主动放弃：

1. 从候选状态中移除押注凭证记录的装备。
2. 按失败或放弃规则计算保留奖励。
3. 清除押注凭证。
4. 一次性提交。

异常退出恢复：

1. 下次启动发现激活押注凭证。
2. 按主动放弃规则移除押注装备，不返还消耗品，不加入局内收益。
3. 清除押注凭证并提交。
4. 主菜单显示一次明确提示，说明上一局因异常中断已按放弃处理。

关闭窗口、崩溃和断电都不依赖退出回调。退出回调只用于尽力刷新日志，不承担规则正确性。

### 事务化 Meta 提交

`Save()` 改为返回结果的内部提交接口。所有公开 Meta 操作遵循：

1. 复制当前 `FGT_MetaProgressState`。
2. 在副本上完成完整业务操作和校验。
3. 将候选状态写入存档。
4. 写盘成功后替换内存 `State`。
5. 写盘失败则保留原内存状态，并返回可显示错误。

结算不再调用多个会自行保存的细粒度函数。新增聚合操作处理整次撤离、失败和开局押注，保证一次业务操作只提交一次。

### 备份与损坏恢复

正式槽仍为 `GraytailMeta`，新增备份槽 `GraytailMetaBackup`。

提交顺序：

1. 当前正式状态有效时先写备份槽。
2. 写入新的正式状态。
3. 正式写入失败时保留原内存状态，并报告失败。

备份槽写入失败时停止本次提交，不继续覆盖正式槽。

加载顺序：

1. 尝试正式槽并校验版本和状态。
2. 正式槽损坏时尝试备份槽。
3. 备份恢复成功后重写正式槽，并向主菜单显示恢复提示。
4. 两个槽都损坏时进入“存档需要处理”状态，不自动创建新存档。
5. 玩家明确确认重置后才创建空状态。

从备份读出但无法重写正式槽时，状态标记为“恢复待落盘”，允许玩家查看菜单但阻止开局和 Meta 消费，直到重试保存成功。

Sanitize 覆盖金币、统计、装备、天赋、仓库、消耗品库存、loadout、重复 ID、未知 ID 和数量范围。修复后的状态必须重新提交；无法安全修复时视为损坏。

### 发布入口探针

Development 打包支持 `-GraytailStartupProbe`：

- 等待本地 PlayerController 创建 HUD。
- 验证主菜单可见。
- 模拟选择教程或标准难度。
- 验证创建了有效运行上下文。
- 使用唯一临时存档槽。
- 输出结构化结果并正常退出。

Shipping 不暴露探针，但必须验证进程稳定启动、Pak 内容正确且无调试镜像。

## PR 2 设计

### 模拟所有权

`UGT_RunSubsystem` 负责持有战斗模拟对象和显式暂停状态。模拟对象不引用任何 Widget。

新增类型：

- `FGT_CombatSimulationState`：玩家和怪物位置、速度、攻击阶段、弹道、计时器、暂停状态。
- `FGT_CombatInputState`：移动方向、攻击请求、逃跑请求。
- `FGT_CombatViewSnapshot`：UI 所需的只读轻量数据，不包含调试字符串。
- `FGT_CombatSimulationConfig`：从游戏数据快照取得的玩法参数。

UMG 只提交输入并读取 ViewSnapshot。伤害、命中、冷却和 AI 决策只能由模拟器产生。

### 固定时间步

模拟步长固定为 `1/60` 秒。

外层 Tick 使用累积器：

- 每帧加入真实 DeltaTime。
- 单帧最多接收 `0.25` 秒，避免恢复焦点后追赶过久。
- 按固定步长循环推进。
- 剩余不足一步的时间留到下一帧。

暂停时不累积时间。恢复后从当前状态继续，不补算暂停期间时间。

### 确定性随机

每次战斗使用从以下内容派生的独立 `FRandomStream`：

- Run Seed
- 房间坐标
- 战斗实例序号

怪物漫游、分裂偏移和攻击扰动不再使用 `FMath::FRand`。纯视觉粒子可以继续使用非确定随机，但不得影响规则位置或伤害。

### 碰撞和伤害

飞弹从“移动后点距离判定”改为上一位置到新位置的线段扫掠，检测线段与玩家碰撞圆。

激光按固定模拟步计时，伤害间隔不依赖渲染帧数。近战阶段允许一次固定步跨越多个阶段时按顺序补齐，不能漏掉命中窗口。

玩家无敌时间、攻击冷却和怪物攻击阶段全部属于模拟状态。

### UI 适配

`UGT_RoomViewWidget` 保留：

- 输入采集。
- 贴图、动画、音效和特效。
- 从 ViewSnapshot 插值显示位置。
- 门和房间表现。

移除：

- 规则伤害。
- AI 决策。
- 弹道命中。
- 攻击冷却裁决。
- 以焦点状态作为暂停规则。
- 每帧构造 `FGT_DebugRunSnapshot` 和摘要字符串。

### 战斗验收

对同一 Seed 和输入时间线分别使用 30、60、144 FPS 以及包含 200ms 卡顿的帧序列驱动外层 Tick，最终规则状态必须一致。

测试覆盖：

- 近战阶段跨帧。
- 单发、散射和激光。
- 飞弹扫掠命中与擦边未命中。
- 分裂怪物。
- 暂停和恢复。
- UI 不存在时模拟仍可运行。
- Widget 重建不重置战斗。

## PR 3 设计

### 玩法数值 JSON 化

玩法数值迁入现有领域 JSON：

- 玩家移动速度、加速度和减速度。
- 玩家攻击冷却、无敌时间和攻击范围。
- 通用飞弹速度与碰撞半径。
- 各难度满压力房间伤害。
- 其他仍会改变胜负、收益或可操作窗口的常量。

纯表现参数不迁入玩法 JSON：

- UI 颜色。
- 面板布局和像素尺寸。
- 金光、闪白、飘字等不影响规则的动画时长。
- 美术帧播放速度。

加载器继续执行严格字段、类型、范围和交叉引用校验。非法配置阻止开局。

### 输入重映射

新增正式输入动作层，由 PlayerController 统一解释动作，再转发给 HUD 或模拟器。

首轮支持键鼠：

- 移动。
- 攻击。
- 搜索/交互。
- 使用道具。
- 地图。
- 暂停。
- 菜单确认和返回。

设置页允许重新绑定，并把映射保存到用户配置。冲突绑定必须提示并要求确认替换。UI 提示从当前绑定动态生成，不再写死 `W/S`、`F`、`Q`。

完整手柄导航和图标提示不在本轮范围。

### 中文本地化

所有玩家可见文本使用 `LOCTEXT`、`NSLOCTEXT` 或 String Table，进入 UE GatherText 流程。

本轮只交付中文源文本，不制作英文翻译。日志、内部 ID 和调试文本不要求本地化。

### 自动验证与 CI

新增一键发行验证脚本，顺序执行：

1. GraytailEditor 构建。
2. 完整命令行冒烟。
3. Development BuildCookRun。
4. Development StartupProbe。
5. Shipping BuildCookRun。
6. Pak 六份 JSON 清单检查。
7. Shipping 启动存活探针。
8. 正式存档与临时槽隔离检查。

GitHub 托管 Runner 没有 UE 5.7，因此轻量 CI 负责：

- JSON 语法和顶层版本检查。
- 禁止 Shipping UI 引用作弊入口的静态守卫。
- 文档和配置一致性检查。
- C++ 格式及可执行的非引擎静态检查。

完整 UE 构建由本机发行脚本执行；未来配置自托管 Windows Runner 后可直接复用同一脚本。

### 文档

更新或归档仍声称 UI、战斗、存档和 Meta 未实现的状态文档。新增：

- 玩家启动与操作说明。
- JSON 数值字段说明。
- 存档恢复行为。
- 发行验证步骤。
- 三层架构说明：数据、模拟、表现。

## 错误呈现

主菜单统一显示以下阻断错误：

- 游戏数据无效。
- 存档版本不支持。
- 正式槽与备份槽均损坏。
- 开局押注写盘失败。
- 上一局异常中断已按放弃处理。

错误必须同时写入日志。可恢复警告允许进入菜单；会破坏进度一致性的错误阻止开局。

## 非目标

- 不实现云存档。
- 不实现局中继续游戏。
- 不实现联网、排行榜或反作弊服务。
- 不实现完整手柄支持。
- 不制作英文翻译。
- 不把纯视觉样式全部迁入 JSON。
- 不把 UMG 战斗重写成 Actor/Component 世界战斗。

## 完成标准

全部三个 PR 完成后：

1. 双击 Shipping 程序即可进入主菜单并开始游戏。
2. Shipping 玩家无法从 UI 或命令行启用 GM 作弊。
3. 强退、崩溃或断电后不能保住本应损失的装备。
4. 存档失败不会让内存状态假装成功。
5. 损坏正式存档可从备份恢复，双损坏不会静默清空。
6. 战斗在无 Widget 环境下可运行并通过确定性测试。
7. 不同渲染帧率不会改变战斗规则结果。
8. 所有影响胜负和收益的目标数值可通过 JSON 修改。
9. 键鼠绑定可修改，界面提示与实际绑定一致。
10. 玩家可见中文进入 UE 本地化采集流程。
11. 一条命令可完成构建、冒烟、Development/Shipping 打包和启动验证。
